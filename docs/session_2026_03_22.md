# Session 2026-03-22 — What was built and fixed

## 1. Force re-translate option

### Problem
Running pipeline "Translate (AI only)" on a mod with all strings already cached → 0 strings translated, no way to force re-translation from UI.

### Solution
Added a **Force re-translate** toggle that bypasses every level of caching.

**UI:**
- `strings.html` — "Force re-translate" toggle (yellow label) in the AI settings panel (⚙ sliders button)
- `mod_detail.html` — "Force" switch directly next to the Run button on pipeline step 3 (Neural Translation)

**Backend wiring — `translate_strings` path (strings page):**
```
UI checkbox → POST options.force
  → jobs.py: _create_translate_strings_job(force=force)
  → workers.py: translate_strings_worker(force=force)
    - skips "untranslated only" filter when force=True
    - passes force=True to translate_texts()
  → translate_texts(force=True) → translate_batch(force=True)
    → bypasses in-memory pipeline/ensemble cache
```

**Backend wiring — `translate_mod` path (pipeline step 3):**
```
UI checkbox → POST options.force
  → jobs.py: _create_translate_mod_job(force=force)
  → workers.py: translate_mod_worker(force=force)
  → esp_engine.cmd_translate(force=force)
  → translate_strings(force=force)
    - skips ".trans.json per-string cache" when force=True
    - passes force=True to translate_texts()
  → translate_texts(force=True) → bypasses in-memory cache
```

**Files changed:**
- `translator/web/templates/strings.html` — checkbox + `options: { force }` in `translateAll()`
- `translator/web/templates/mod_detail.html` — Force switch + `force` in `runStep('translate')`
- `translator/web/routes/jobs.py` — extract `options.force`, pass to both factory functions
- `translator/web/workers.py` — `translate_strings_worker(force=)` and `translate_mod_worker(force=)`
- `scripts/esp_engine.py` — `cmd_translate(force=)`, `translate_strings(force=)`, pass to `translate_texts`

## 2. Silent failure bug — English originals saved as translations

### Problem
When the Mac remote server (192.168.1.220:8765) becomes unreachable mid-job, `RemoteBackend.translate()` catches the `ConnectTimeout` and calls `results.extend(batch)` — returning the original **English text** as the translation. This gets saved to cache as `status=translated` with high quality scores (English vs English = ~100%).

Observed in logs: batches 760, 764, 768, 772, 776, 780, 784 all logged `"RemoteBackend: batch N failed — returning originals"` while the job still reported 812/812 complete.

Same bug existed in `translate_batch()` fallback in `esp_engine.py`.

### Fix
Return empty strings on failure → `translate_texts` treats empty translation as untranslated/skipped → string stays pending in cache.

**`translator/models/remote_backend.py`:**
```python
# Before
except Exception:
    log.exception("RemoteBackend: batch %d failed — returning originals", i)
    results.extend(batch)

# After
except Exception:
    log.exception("RemoteBackend: batch %d failed — skipping (empty strings)", i)
    results.extend([""] * len(batch))
```

**`scripts/esp_engine.py` — `translate_batch()`:**
```python
# Before
except Exception:
    log.exception("translate_batch error")
    return list(texts)

# After
except Exception:
    log.exception("translate_batch error")
    return [""] * len(texts)
```

**Why empty strings are safe:**
- `translate_strings_worker`: `if r["skipped"] or not r["translation"]: continue` — skips saving
- `translate_strings`: `if not t: continue` — skips setting translation on the string object
- Result: failed batches leave strings as untranslated (pending), not corrupted with English

**Recovery:** Strings that were saved with English text before this fix can be identified in the UI (translation == original). Re-run with Force re-translate enabled to fix them.
